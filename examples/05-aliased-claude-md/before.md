# CLAUDE.md — Data Development Project Project Context

## Project Overview

Data Development Project is a multi-tenant restaurant analytics platform. We ingest sales data from Soviet POS systems via SFTP, normalize and transform it through a series of Edge Functions, materialize it into denormalized analytical views, and serve dashboards to restaurant owners. The system is designed to onboard any establishment without code changes by reading per-establishment configuration from the `establishment_settings` JSONB column on the `establishments` table.

This is a development project. We are building the onboarding system. We are not actually onboarding restaurants in production. Our current test establishment is Fred's Italian Bistro, but Fred's Italian Bistro's data is expendable test data that we destroy and recreate as needed during development.

## Architecture

The architecture has the following components:

- **n8n** orchestrates all workflows. Every multi-step process runs in n8n. The n8n instance lives at our Hostinger VPS at workflows.example.com. n8n triggers Edge Functions, monitors progress, detects stuck states, and restarts the orchestrator if needed.
- **Supabase Edge Functions** handle heavy data processing. They read from Soviet SFTP, parse CSV files, normalize records, and write to the database. Edge Functions are deployed via `supabase functions deploy`.
- **PGMQ** (Postgres Message Queue) manages job queues for the Edge Functions. The Edge Functions enqueue jobs and workers dequeue and process them.
- **pg_cron** schedules recurring jobs that the workers process. Currently we have pg_cron jobs for materialized view refreshes, daily report generation, and the customer satisfaction report aggregation.
- **The materialized views** are the main analytical surface. The materialized views are refreshed concurrently using `REFRESH MATERIALIZED VIEW CONCURRENTLY`. The refresh policy is documented in `docs/mv-refresh.md` and is also encoded in the `obs_mv_refresh_policy` table.
- **The admin dashboard** is a Next.js app at admin.example.com that shows real-time migration progress via WebSockets and surfaces the customer satisfaction report and the wine report to restaurant owners.

## Multi-Tenant Architecture — Zero Hard-Coding

This is critical. NEVER hard-code establishment-specific values anywhere in the codebase. Every establishment-specific value MUST come from the `establishment_settings` JSONB column on the `establishments` table.

This includes:
- The `establishment_id` (UUID)
- The establishment name
- The Soviet POS account ID for that establishment
- The SFTP base path for that establishment (e.g., `/34921/`)
- The SFTP credentials key for that establishment
- Business hours and operating schedule for that establishment
- Menu structure for that establishment
- Any pricing rules for that establishment
- Any business rules specific to that establishment (e.g., grace period for menu items)

All queries against any table must filter by `establishment_id`. All Edge Functions must accept `establishment_id` as a parameter. All n8n workflow variables must use the workflow input `establishment_id`, never a hard-coded value. The customer satisfaction report and the wine report are both per-establishment and both must filter by `establishment_id`.

Row-level security policies enforce data isolation at the database level, but this is a defense-in-depth measure, not a substitute for application-level filtering.

## Branch and Worktree Discipline

We follow strict branch discipline. One working tree equals one active branch. The current branch must be confirmed before any file edits. If the user's request is unrelated to the current branch's feature, the agent must stop and ask the user to choose: (A) continue on the same branch if the work truly belongs there, (B) switch to a new branch in this folder, or (C) use `git worktree add` to create a parallel working directory for the unrelated work.

Red flags that always trigger this stop include: topic jumps (e.g., asking about the wine report while the current branch is about materialized view refreshes), "quick unrelated fixes," and multiple unrelated deliverables in one chat thread.

The agent must never silently work across unrelated features in one branch.

## Reports

There are two main reports the system generates:

### The Wine Report

The wine report is a per-establishment report that summarizes wine sales over a configurable date range. It is generated by the Edge Function at `supabase/functions/wine-report/index.ts`. The Edge Function reads from the materialized view `mv_wine_sales` and aggregates by varietal, region, and price tier. The wine report is delivered via email to the establishment owner every morning at 6am Eastern, scheduled via pg_cron job `wine_report_daily`. The wine report respects the establishment's operating schedule and excludes closure days from the average calculations. The wine report is also surfaced in the admin dashboard.

### The Customer Satisfaction Report

The customer satisfaction report is a per-establishment report that aggregates customer feedback from multiple sources (Soviet tips data, Yelp API, Google Reviews API) and produces a rolling 30-day satisfaction score. It is generated by the Edge Function at `supabase/functions/customer-satisfaction-report/index.ts`. The customer satisfaction report writes to the materialized view `mv_customer_satisfaction` which is refreshed nightly via pg_cron job `customer_satisfaction_refresh`. The customer satisfaction report is also delivered via email to the establishment owner weekly on Monday mornings at 9am Eastern.

Both reports must filter by `establishment_id` at every layer.

## Development Workflow Philosophy

Build the system. Do not do the work the system should do. If you find yourself manually triggering Edge Functions, manually running SQL queries to fix data issues, or manually restarting stuck processes — STOP. Identify why the automation didn't handle it, and fix the automation. The exception is ONE test call to verify a fix works. Multiple manual runs means you're doing the work instead of building the system.
